Relatorio tecnico da AgroSmart API

Leitura integral da branch dev, com arquitetura, funcoes, endpoints, integracoes, banco, deploy e riscos.

Base: commit 8909debb9b90989b3407da08c33846064e7d63ac em 18/06/2026. Repositorio: GitHub / dev.

Resumo tecnico

  • Objetivo: coletar cotacoes de graos por scraping, guardar snapshots no PostgreSQL e oferecer consultas atuais, historicas, filtradas, comparativas e CSV por uma API REST.
  • Stack: Node.js CommonJS, Express, Puppeteer, PostgreSQL/Neon, node-cron, Swagger UI e OpenAPI 3.0.3.
  • Fontes ativas: Coamo, C.Vale e LAR. O arquivo Granos existe, mas nao participa da aplicacao.
  • Arquitetura: rotas finas chamam services; services coordenam cache, scraping e regras; repositories executam SQL; utils normalizam, filtram e geram relatorios.
  • Producao: a URL publicada aponta para Render e respondeu em /health e /docs. O OpenAPI publicado coincide com o arquivo local, mas isso nao prova que todo o codigo implantado veio do mesmo commit.
  • Estado geral: a separacao de responsabilidades e boa para o tamanho atual, mas faltam testes, migracoes, CI, controle de frescor, observabilidade e garantias para scheduler em multiplas instancias.

O fluxo principal funciona, mas ha riscos de consistencia e operacao

A API preserva o ultimo dado valido quando o scraping falha, reduz chamadas repetidas com cache e evita duas coletas simultaneas da mesma fonte dentro do processo. Em contrapartida, pode devolver dados antigos sem informar a idade, depende de paginas sujeitas a bloqueios e executa o scheduler dentro do servidor web.

Pontos fortes

  • Queries SQL parametrizadas.
  • Comparacao de token com timingSafeEqual.
  • Retry, cache e deduplicacao de coleta por fonte.
  • Historico e ultimo snapshot separados.
  • Erros HTTP padronizados com AppError.
  • OpenAPI amplo e Swagger publicado.

Principais alertas

  • Comparativo inclui C.Vale na resposta, mas nao no calculo da melhor fonte.
  • Scheduler pode duplicar coletas em varias replicas.
  • Scraping falho pode esperar minutos por retries inadequados.
  • saveSnapshot nao usa transacao.
  • README, package e codigo estao fora de sincronia.
  • npm audit encontrou 11 dependencias vulneraveis.

Fluxo de uma consulta atual

Cliente
  -> Express + requestLogger
  -> authenticateToken em /api/cotacoes/*
  -> rota
  -> service
     -> cache em memoria, se disponivel
     -> senao, cotacoes_ultima no PostgreSQL
     -> senao, Puppeteer no site da fonte
        -> valida array nao vazio
        -> INSERT em cotacoes_historico
        -> UPSERT em cotacoes_ultima
        -> atualiza cache
  -> resposta JSON ou CSV
  -> log resumido da requisicao em logs_requisicoes

Arquitetura e inicializacao

O sistema usa uma arquitetura em camadas simples. Nao ha framework de injecao, ORM ou container: modulos CommonJS importam dependencias diretamente.

CamadaResponsabilidadeDependencias principais
Entrada HTTPExpress, JSON, CORS, health, docs e montagem das rotas.src/app.js
MiddlewaresAutenticacao, log de requisicao, 404 e tratamento de erros.src/middlewares/
RotasLe query strings, chama services e define formato HTTP.src/routes/
ServicesOrquestra cache, coleta, retry, filtros, comparacao, historico e exportacao.src/services/
ScrapersAutomatiza paginas dos fornecedores e extrai tabelas.src/scrapers/, Puppeteer
PersistenciaCria estrutura e executa consultas PostgreSQL.src/database/, pg
UtilitariosDatas, normalizacao, preco numerico, filtros, CSV e series historicas.src/utils/
AgendamentoDispara coletas em dois horarios por cron.src/jobs/, node-cron

Sequencia de subida

  1. dotenv carrega o ambiente e o OpenAPI e carregado em memoria.
  2. Middlewares e rotas sao registrados.
  3. validateAuthConfig() bloqueia a subida se autenticacao estiver ativa sem token.
  4. initDatabase() cria tabelas e indices com CREATE IF NOT EXISTS.
  5. bootstrapCotacoesCache() carrega o ultimo snapshot das tres fontes.
  6. Express escuta em PORT.
  7. O scheduler registra os cron jobs no mesmo processo.
Consequencia: banco e autenticacao sao obrigatorios para o processo subir. Falha no banco encerra o processo com codigo 1.

Banco de dados e modelo de persistencia

O sistema usa PostgreSQL diretamente. A API cria o schema na inicializacao, sem ferramenta de migracao. Dados de cotacao ficam em JSONB, preservando o formato retornado pelos scrapers.

TabelaFinalidadeCampos principaisIndice
cotacoes_historicoUm registro por coleta bem-sucedida.id, fonte, dados_json, quantidade_itens, coletado_em, janela_horario, tipo_disparo, criado_em(fonte, coletado_em DESC)
cotacoes_ultimaUm registro por fonte para leitura rapida.fonte PK, dados_json, quantidade_itens, coletado_em, janela_horario, tipo_disparo, atualizado_emChave primaria em fonte
logs_requisicoesResumo de chamadas HTTP concluidas.id, metodo, rota, status_code, duracao_ms, criado_em(criado_em DESC)

Comportamento de cache

  • Cache e um objeto em memoria por processo: coamo, cvale e lar.
  • Na subida, o cache recebe cotacoes_ultima.
  • Sem cache e sem banco, a primeira consulta dispara scraping.
  • Nao existe TTL, idade maxima ou indicador de dado obsoleto.
Consistencia: historico e ultima cotacao sao gravados em duas queries sem transacao. Se a segunda falhar, o historico fica atualizado e cotacoes_ultima nao.

Execucao, instalacao e deploy

Local

  • npm install: instala dependencias e executa o postinstall do Chrome.
  • npm run dev: usa node --watch src/app.js.
  • npm start: usa node src/app.js.
  • npm test: termina com erro proposital, pois nao ha testes configurados.

Producao observada

  • Host publicado: https://agrosmartapi.onrender.com.
  • Cabecalho x-render-origin-server: Render confirma Render como origem.
  • Cloudflare aparece como servidor de borda.
  • Nao existe Dockerfile, render.yaml, workflow de CI/CD ou script de deploy.
  • Build command, start command, plano, health check e variaveis sao gerenciados fora do repositorio.
Scheduler embutido: em scale-to-zero os jobs nao rodam durante sleep; em duas ou mais replicas, cada replica registra os mesmos jobs e pode gravar snapshots duplicados. A deduplicacao inFlight funciona apenas dentro de um processo.

Seguranca e auditoria de dependencias

Controles existentes

  • Token obrigatorio por padrao nas rotas de negocio.
  • Suporte a Bearer e x-api-token.
  • Comparacao temporalmente segura para tokens do mesmo tamanho.
  • SQL parametrizado nos repositories.
  • .env ignorado pelo Git.

Lacunas

  • Token unico e estatico, sem identidade, escopo, expiracao ou rotacao implementada.
  • CORS aberto para qualquer origem.
  • Sem rate limit; force=true pode iniciar browser e gerar carga cara.
  • Sem Helmet/cabecalhos endurecidos.
  • Detalhes de falha de scraper podem expor causa interna na resposta 502.
  • SSL do PostgreSQL nao valida certificado quando ativado.

Limites

  • Nao foi usado token de producao nem executado endpoint protegido.
  • Nao foram acessados dados do Neon ou configuracao interna do Render.
  • Os scrapers nao foram executados ponta a ponta para evitar coleta externa e gravacao indireta; foram lidos integralmente e os sites receberam apenas probes HTTP simples.
  • O health atual nao verifica banco, Chrome, scheduler ou paginas de origem.

Questoes abertas para a equipe

  4. Qual e a idade maxima aceitavel para uma cotacao ser chamada de atual?
  5. O comparativo deve comparar precos entre locais diferentes ou exigir o mesmo local/unidade?
  6. Logs e historico precisam de politica de retencao?
  7. A branch dev deve ser protegida no GitHub?
